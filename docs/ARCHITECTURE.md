# Architecture

## Overview

`openclaw-lmstudio` connects OpenClaw to one of two LLM backends:

1. **LM Studio** (local) — Free, private, runs on your GPU/CPU
2. **Claude** (Anthropic API) — Cloud-based, best-in-class reasoning

The `PROVIDER` environment variable controls which backend is used.

## Component Diagram

```
Host Machine
├── LM Studio (localhost:1234)          ← PROVIDER=lmstudio
│   ├── OpenAI-compatible API
│   ├── GPU/CPU inference
│   └── Model management UI
│
├── Anthropic API (api.anthropic.com)   ← PROVIDER=claude
│   └── Claude models via ANTHROPIC_API_KEY
│
├── Docker Engine
│   └── openclaw-lmstudio container
│       ├── OpenClaw agent (Node.js)
│       ├── openclaw.json (provider config, generated by setup.sh)
│       ├── /workspace (bind mount → host project)
│       └── /home/node/.openclaw (config persistence)
│
└── Workspace Directory (your code)
    └── Bind-mounted read-write into container
```

## Networking

The container reaches LM Studio on the host via `host.docker.internal`, which Docker Desktop resolves automatically. On Linux, the `extra_hosts` directive in `docker-compose.yml` maps it to the host gateway.

```
Container → host.docker.internal:1234 → LM Studio API
```

The OpenClaw gateway is exposed only on `127.0.0.1:18789` (loopback), not on all interfaces.

## Security Layers

| Layer | Mechanism |
|-------|-----------|
| Capabilities | All dropped, `NET_BIND_SERVICE` added back |
| Privileges | `no-new-privileges: true` |
| Resources | 2GB RAM, 2 CPUs, 512 PIDs max |
| User | Non-root `node` (uid 1000) |
| Network | Gateway port published only to `127.0.0.1`; agent sandboxes have `network: none` |
| Workspace | Bind mount with explicit `rw` — container cannot access other host directories |

## Configuration Flow

```
.env (user settings)
  ↓
scripts/setup.sh (reads .env, generates openclaw.json, reports enabled deps)
  ↓
openclaw.json (provider + gateway config)
  ↓
Dockerfile (bakes openclaw.json into image at /opt/openclaw-seed/;
            always installs jq/Homebrew/gogcli/gh;
            conditionally installs Chromium, ffmpeg via INSTALL_* build args)
  ↓
scripts/start.sh (seeds config on first run, installs runtime deps if enabled,
                   validates provider connectivity, launches OpenClaw gateway)
```

Config is seeded from the image into `.openclaw-files/.openclaw/` on first run. The gateway then
manages its own runtime config (auth tokens, etc.) in that directory. To apply config
changes, clear `.openclaw-files/.openclaw/` and rebuild the image.

## Optional Dependencies

Skills rely on external tools not in the base image. A two-tier install approach keeps
the default image lean while allowing opt-in:

| Tier | Flags | Install Method | Requires Rebuild? |
|------|-------|---------------|-------------------|
| Always included | — | `apt-get` + Homebrew in Dockerfile | N/A (always present) |
| Build-time | `INSTALL_CHROMIUM`, `INSTALL_FFMPEG`, `INSTALL_QMD` | `apt-get` / Bun in Dockerfile | Yes |
| Runtime | `INSTALL_GO`, `INSTALL_UV`, `INSTALL_NPM_GLOBALS` | Downloaded in `start.sh` on first boot | No |

Runtime deps persist on the `.local` volume. All installers are idempotent and non-fatal.

## File Descriptions

| File | Purpose |
|------|---------|
| `Dockerfile` | Extends official OpenClaw image; always installs jq, Homebrew, gogcli, gh; conditionally installs Chromium, ffmpeg |
| `docker-compose.yml` | Orchestration with volumes, networking, security, resource limits, and dep flags |
| `openclaw.json` | OpenClaw provider config (generated by `setup.sh`) |
| `.env` / `.env.example` | User-configurable settings: provider, model, workspace, optional dependencies |
| `scripts/setup.sh` | One-command setup: generates config, reports enabled deps, builds image, starts container |
| `scripts/start.sh` | Container entrypoint: installs runtime deps, validates provider, launches gateway |
| `scripts/copy-config.sh` | Migrates existing `~/.openclaw` config into `.openclaw-files/` |
